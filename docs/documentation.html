1. Vue d'ensemble du projet

Boosterfish est une application mobile de référencement et découverte de spots de pêche côtière au Maroc et dans le monde. Elle combine une carte interactive avec 6200+ spots, des données de marées temps réel, une encyclopédie de poissons, des techniques de pêche et un réseau de magasins de pêche géolocalisés.

🎯 Objectifs produit

  • Localiser rapidement les meilleurs spots de pêche
  • Obtenir les conditions maritimes en temps réel
  • Apprendre les techniques adaptées à chaque poisson
  • Trouver du matériel près de son spot
  • Partager et apprendre au sein d'une communauté

🏗️ Stack technique

  • Framework : Flutter 3.x (Dart 3)
  • Carte : flutter_map + OpenStreetMap
  • Marées : prévisions Open-Meteo publiées par le backend Firebase
  • Géolocalisation : geolocator
  • Persistence : path_provider + JSON cache
  • Deep linking : url_launcher

L'application adopte une architecture Clean Architecture simplifiée avec séparation claire entre Models, Services, Pages et Widgets réutilisables. Aucun state manager externe n'est utilisé — la gestion d'état repose sur setState et des patterns de cache optimisés.

2.2 Couche Models (Données)

ModèleFichierResponsabilité
Spot lib/models.dart Représente un spot de pêche avec nom, coordonnées GPS, type (sable/roche/falaise/mixte/remote), espèces de poissons, notes. Sérialisation JSON/CSV + inférence du type par analyse sémantique du nom.
SpotType lib/models.dart Enum 5 valeurs avec extension Dart : couleur associée (#E74C3C, #9B59B6, #0066CC, #E67E22, #1ABC9C) et label localisé. Clé pour le système de légende interactive.
FishingShop lib/models/fishing_shop.dart Magasin de pêche : nom, coordonnées, téléphone, adresse, image, horaires, tags (Appâts, Cannes, Moulinet…), note. Sérialisation JSON/CSV.
ShopSpotGroup lib/models/fishing_shop.dart Regroupement logique : associe un spot à la liste des magasins situés à moins de 50 km. Utilisé pour l'affichage par section dans ShopsPage.
TidePoint lib/models/tide_data.dart Point temporel d'une courbe de marée : DateTime + hauteur (mètres).
TideData lib/models/tide_data.dart Données complètes marées : liste horaire, low/high/next, localisation, + données astronomiques (AstroData). Fallback intégré.
AstroData lib/services/astronomy_service.dart Phase lunaire, coefficient, activité poisson (0-1), lever/coucher soleil & lune, transit solunaire major/minor. Calculée algorithmiquement sans API externe.

Inférence du type de spot (SpotType._inferSpotType)

Analyse sémantique du nom du spot pour classifier automatiquement :

  • Plage de sable : mots-clés "plage", "sable", "aftas"
  • Spot rocheux : "roch", "حرش", "roche", "pointe", "cap", "isk"
  • Mixte sable/rocher : présence des deux familles
  • Falaise : "falaise"
  • Remote : défaut

2.3 Couche Services (Logique métier)

ServiceFichierFonctionnalités
SpotService lib/main.dart Chargement CSV depuis assets + cache JSON persistant via path_provider. Parsing asynchrone avec compute() sur isolate. Gestion des lignes malformées (skip graceful).
TideService lib/services/tide_service.dart Lecture des prévisions publiées dans Firestore par le backend. Le backend utilise les endpoints Open-Meteo client avec une clé commerciale conservée côté serveur. L'application ne contacte jamais Open-Meteo directement ; extraction LOW/HIGH/NEXT et états sans données explicites.
AstronomyService lib/services/astronomy_service.dart Calculs astronomiques 100% offline :
  • Phase lunaire via synode (29.53j) depuis référence 6 jan 2000
  • Coefficient basé sur amplitude marée / 3.8m × 120
  • Lever/coucher soleil via déclinaison saisonnière
  • Lever/coucher lune via retard 50min/jour
  • Transit solunaire major/minor
  • Score d'activité poisson : coeff(40%) + phase(30%) + proximité transit(30%)
ShopService lib/services/shop_service.dart Chargement CSV + cache JSON. Regroupement géospatial : associe chaque magasin au spot le plus proche (rayon 50km) via calcul Haversine (package latlong2). Tri par densité de magasins.

2.5 Navigation & App Shell

L'application utilise un AppShell avec IndexedStack pour préserver l'état de chaque onglet. La navigation se fait via :

  • BottomNavigationBar custom : 5 onglets (Home, Fish, Add, Map, Settings) avec bouton central "Add" en gradient
  • GlobalKey (appShellKey) pour navigation programmatique depuis n'importe quel widget
  • MaterialPageRoute pour les pages secondaires (Techniques, Communauté et Magasins)
class AppShell extends StatefulWidget { static const List _pages = [ HomePageWrapper(), // Index 0 SpeciesPageWrapper(), // Index 1 AddSpotPlaceholder(), // Index 2 SpotFinderPage(), // Index 3 SettingsPageWrapper(), // Index 4 ]; } // Navigation depuis HomePage vers une page externe : void _goTo(BuildContext context, Widget page) { Navigator.of(context).push(MaterialPageRoute(builder: (_) => page)); }

2.6 Gestion d'état & Caching

MécanismeDescription
JSON CacheLes spots et magasins sont persistés en JSON dans getApplicationDocumentsDirectory(). Chargement rapide au démarrage, re-synchronisation CSV uniquement si cache vide.
Isolate ComputingParsing CSV/JSON via compute() sur isolate séparé (mobile/desktop). Fallback thread principal sur web.
MarkerCacheManagerCache de widgets SpotMarker par clé de spot et d'état de sélection. Évite les rebuilds inutiles lors du zoom/pan.
Debounced Map UpdatesTimer à 100ms sur les événements de carte pour éviter les recalculs excessifs des spots visibles.
AutomaticKeepAliveMapScreen utilise AutomaticKeepAliveClientMixin pour conserver l'état de la carte lors du changement d'onglet.
RefreshIndicatorHomePage supporte le pull-to-refresh pour recharger les données de marées.

8. Spécifications Techniques Avancées

Optimisations de performance

  • Isolate computing : parsing CSV/JSON déporté sur isolate via compute()
  • Marker cache : MarkerCacheManager avec clé de spot et d'état de sélection
  • Visible bounds filtering : seuls les spots dans la viewport sont rendus
  • Debounced updates : 100ms de debounce sur les événements carte
  • AutomaticKeepAlive : la carte conserve son état lors du switch d'onglet
  • Debug mode silencieux : debugPrint désactivé en release

Sécurité & Robustesse

  • Fallback intégré sur toutes les API (marées, astronomie)
  • Gestion gracieuse des lignes CSV malformées (skip + log)
  • Timeout HTTP : 10 secondes
  • Permissions géolocalisation gérées (denied, deniedForever)

Architecture de la carte (MapScreen)

  1. FlutterMap avec MapController pour contrôle programmatique
  2. TileLayer OpenStreetMap avec 3 sous-domaines (a,b,c)
  3. MarkerLayer conditionnel pour la position GPS
  4. SpotsCanvasLayer pour le rendu optimisé des 6200+ spots
  5. ClusterIndex pour le regroupement spatial

Formule du score d'activité poisson

score = (coefficient / 120) × 0.40 # Amplitude marée + (1 - |phase - 0.5| × 2) × 0.30 # Phase lunaire + (1 - distTransit / 720) × 0.30 # Proximité transit label = score ≥ 0.75 ? "Excellente" : score ≥ 0.55 ? "Bonne" : score ≥ 0.35 ? "Moyenne" : "Faible"
